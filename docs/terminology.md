# NSN Terminology Guide

## Purpose

This document defines approved user-facing terminology for NSN. Reference it when adding new UI, copy, documentation, onboarding flows, tests, or contributor guidance so wording stays consistent across the project.

Use this guide as the source of truth for preferred terms, deprecated language, and the reasoning behind naming choices.

## Core Principles

- Use welcoming, human language.
- Avoid implying guarantees of safety, identity verification, or moderation authority.
- Prefer prototype-aware wording during alpha.
- Keep terminology consistent across app screens and documentation.
- Avoid copy that makes app confidence feel like a requirement for joining the community.

## Approved Terms

### Readiness Preview

- Preferred term: Readiness Preview
- Deprecated/avoid terms: Safety Check, Safety Score, Verification Status, Trust Rating
- Reasoning: Describes a lightweight preview of preparedness without implying that NSN can guarantee safety, verify identity, or certify trustworthiness.

### Host Details Preview

- Preferred term: Host Details Preview
- Deprecated/avoid terms: Host Verification, Host Approval, Vetted Host, Certified Host
- Reasoning: Frames host information as a preview during the alpha prototype, not as a formal verification or endorsement process.

### Prototype Contact Preview

- Preferred term: Prototype Contact Preview
- Deprecated/avoid terms: Emergency Contact, Verified Contact, Safety Contact, Trusted Contact
- Reasoning: Makes clear that contact-related behavior is part of the prototype experience and should not be read as emergency infrastructure or verified support.

### Shared Preview Visibility

- Preferred term: Shared Preview Visibility
- Deprecated/avoid terms: Public Safety Sharing, Community Monitoring, Live Tracking, Moderator View
- Reasoning: Communicates that selected preview information may be visible in a shared context without implying live surveillance, moderation authority, or continuous tracking.

### Local Report Preview

- Preferred term: Local Report Preview
- Deprecated/avoid terms: Incident Report, Police Report, Safety Report, Official Complaint
- Reasoning: Keeps reporting language prototype-aware and avoids implying that NSN creates official, legal, or emergency reports.

### Boundary Options

- Preferred term: Boundary Options
- Deprecated/avoid terms: Rules, Restrictions, Safety Controls, Enforcement Settings
- Reasoning: Encourages calm, personal language around user preferences without suggesting that NSN enforces behavior or controls other people.

### Community Host

- Preferred term: Community Host
- Deprecated/avoid terms: Verified Host, Official Host, Moderator, Safety Officer
- Reasoning: Describes the person hosting a local gathering in a warm, human way without implying special authority, identity verification, or moderation responsibility.

### Alpha Prototype

- Preferred term: Alpha Prototype
- Deprecated/avoid terms: Production App, Safety Platform, Verified Network, Final Product
- Reasoning: Sets accurate expectations that NSN is still in an early pilot stage and that language should remain careful, provisional, and prototype-aware.

### Alternative Access Pathways

- Preferred terms: Alternative Access Pathways, Assisted RSVP, Host-Assisted Onboarding, Website View, Email Contact
- Deprecated/avoid terms: Just Use The App, App-Only Access, App Required, Self-Service Only
- Reasoning: The app is the primary NSN experience, but it should not be the only door into the community. Some people may need support from a host, family member, community partner, email, a simple web page, WhatsApp/SMS-style contact, printed information, libraries, support groups, or local organisers. Copy should treat these routes as valid and practical, not as awkward exceptions.

## Historical Naming

- NSN (North Shore Nights): Current Sydney/North Shore alpha pilot.
- SoftHello: Future umbrella or global direction for the broader concept.
- SoftShore: Historical exploration name; avoid using it for current product-facing copy unless discussing project history.

## Contributor Guidance

When introducing new terminology:

- Check this document first.
- Prefer existing terminology when it covers the intended meaning.
- Avoid new terms that imply guarantees of safety, identity verification, official reporting, moderation authority, or emergency support.
- Update this document if a new term becomes an approved standard.
